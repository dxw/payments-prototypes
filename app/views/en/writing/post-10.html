Writing about these prototypes

Banking API for human eyes

Both the check balance app and debt prevention letter are made possible with the NatWest Bankline API. It enables software the team builds to automatically check if someone has paid some or all of their tax bill. It’s essential for any payment tools, be them for WRA staff or external users.

Currently the banking API happens entirely in the backend. To view the data now, someone would need to open the terminal to see the raw software running.

So the team asked what a human interface include?

View and search bank transactions

Quite obvious, but being able to clearly view bank transactions, rather than downloading a spreadsheet, which is how it’s currently done.

So first we imagine a homepage, with most recent bank transactions. Divided by matched and unmatched. More about unmatched transactions in the next section.

To accompany this, we had a search feature, with pages of results.

Unmatched bank transactions are a cause of tax debt. As mentioned with redesigning the payment slip, when someone pays the WRA, if the bank reference can’t be matched to a balance, the system assumes they haven’t paid. Which in turn can and does lead to missed deadlines, fines and interest to pay.

Currently bank transaction details are uploaded by spreadsheet to WRA accountancy system, Finance & Operations [F&O]. This includes transactions with the wrong reference, which won’t get matched correctly. Also, given F&O is a Microsoft system, its difficult to amend data once its in there.

So an idea the team had was to make a before step, where unmatched transactions can be highlighted and corrected, before being pushed to F&O.

Highlight transaction

Simply, the homepage idea has transactions group by unmatched, matched or all.

Assuming unmatched transactions are the most important, they sit first. The table shows what is stopping the transaction being matched. Whether that's the reference, account name or amount.

In the prototype, someone can click into the unmatched transaction, and view the original data in detail. This makes sense as someone can edit some parts and you might want to audit.

Edit transaction to match it

An obvious feature to couple with highlight unmatched transactions, is the ability to edit information to match it to a balance. In the examples, someone can both edit an incorrect reference or incorrect account name. With the edited transaction then matched to a balance.
