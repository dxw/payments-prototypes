Writing about these prototypes

Testing MVP with staff

In this third sprint, we tested with staff for the first time. We wanted to test a prototype that would be feasible to build in production code in the following sprint. So what we prototyped was more minimal.

Most common case

Few cases are complicated. Most are just tax, with maybe a little bit of interest. So that’s what we prototyped in this sprint to test with staff. We wanted to test whether they could interpret the figures correctly.

Complex case box

As we were just testing the simpler, most common cases, we dropped the box component at the top of the screen, which highlights that a case is complex.

The single view design still included:

  1. Name and UTRN
  2. Big total box
  3. Subtotal table

Balance time removed

As we only wanted to prototype and test what our developers could realistically build in a week or two, we dropped the timeline component.

As a team, we felt it would be quite time consuming to build in production code. It might be novel, but it wasn't clear to us that it was absolutely necessary. If it came out in research staff need more detail on the provenance of financials, then we'd revisit the idea.

Stretch the design with some complexity

We also tested a slightly more complex case - of someone who has got a penalty and begun to pay off the debt in instalments.

If there’s time: Preview and send PDF

As we wanted to stay focused on a MVP, we decided we’d only show staff the Preview and send the PDF feature, if there was enough time at the end of each user research session.

Navigate from search bar

Both user journeys in testing, started with the search bar component, where staff try enter a unique reference number.

The next post discusses what we learned in the first round of testing and the design changes we made as a result.
